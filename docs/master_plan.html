Executive Read

SwingCoach is past the "can I get videos into an app?" stage. The product now needs a stricter definition of what it means to analyze a swing. The core answer: a detected swing should become a typed evidence bundle, and every feedback claim should trace back to a visible checkpoint, metric, or confidence-gated detector output.

Capture and Library Good MVP bones
Detector Promising V2, not solved
Backend Pipeline shell only
Annotations UI exists, evidence missing
Coaching Needs product brain

Recommended next big milestone

Build Coaching Pipeline V1 around a small, reliable annotation set: address plane, takeaway checkpoint, top checkpoint, delivery shaft checkpoint, head/hip reference, and a confidence-gated immediate feedback card. Defer broad biomechanical metrics until the evidence contract is proven on fixtures and real phone videos.

What "full coaching pipeline" should mean first

Not a full golf scientist in the cloud. First it means: upload a swing, find phases, draw the two or three most useful coach marks, produce one or two priority findings, explain uncertainty, and suggest a drill tied to the finding.

What to avoid

Do not chase dozens of metrics from weak monocular evidence. Do not publish 3D-looking numbers before calibration is clear. Do not anchor product strategy to any one coach or creator. Borrow the general coaching pattern: clear checkpoints, simple visual proof, cause/effect, and a drill constraint.

Architecture Direction

The app should stop treating "analysis" as one blob. A detected swing should become a durable product object with source video, event timings, evidence tracks, coach claims, generated assets, manual edits, and user feedback over time.

Proposed domain objects

SavedSwing SwingWindow AnalysisRun EvidenceBundle CoachFinding

SavedSwing is the user's clip. SwingWindow is the detected address-to-finish range plus phase timestamps. EvidenceBundle stores pose, club, ball, event, and confidence data. CoachFinding is the human-facing claim that references evidence and a drill.

Why this matters

It lets the UI say "we are confident about shaft plane but not hip rotation" instead of "analysis succeeded" or "analysis failed." It also makes it possible to re-run only a metric model, only an annotation renderer, or only the coaching voice when those modules improve.

1

Capture or import

Produce a playable source and one or more candidate swing windows. User can trim or accept.

2

Normalize the swing

Extract frames, infer fps/source time scale, find address/top/impact/finish, and save a versioned event timeline.

3

Build evidence tracks

Run only the evidence sources needed for the selected analysis mode: immediate, standard, deep dive, or research.

4

Generate coach findings

Pick a small number of findings, each backed by a visual checkpoint, confidence, and a drill progression.

5

Render overlays and practice loop

Client-render JSON tracks over clean video, let users save manual notes, then compare future swings against the same finding.

How To Use A Detected Swing

Detection is not the end product. It is the entry point into a sequence of increasingly expensive interpretations. The app should offer immediate feedback quickly and reserve deeper analysis for explicit user intent or background processing.

Now

Clip selection

Use detector ranges to preselect candidate clips in Trim or Auto capture. Let the golfer correct boundaries.

Next

Immediate feedback

Return a fast card: swing found, confidence, rough phase timings, one visible checkpoint, and "worth deep dive?"

Milestone

Deep dive

Run pose/club/ball/event fusion, generate overlays, compute safe metrics, and produce a prioritized coaching plan.

Rule for confidence

Confidence should be first-class. If the ball is occluded, the club shaft is blurred, the camera view is off, or the model cannot infer depth, the app should say so and narrow the feedback instead of pretending to know.

Coaching Pipeline

A useful V1 should be checkpoint-led, not metric-led. The best social swing reviews often freeze one frame, draw one or two strong references, explain what that visible fact causes later, and prescribe one constraint drill.

Stage V1 action Evidence required Output
1. Intake Classify view, handedness, source timing, camera quality, ball visibility, and golfer framing. Video metadata, object detections, simple pose/person box. Analysis mode and warnings.
2. Events Find address, takeaway, lead-arm-parallel, top, delivery, impact estimate, finish. Object detector plus pose/club tracker and temporal smoothing. Versioned event timeline.
3. Evidence Compute only high-trust facts: shaft plane, clubhead relative to hands, head/hip reference, tempo. Per-layer confidence and visible frame evidence. EvidenceBundle JSON.
4. Findings Rank one to three issues by severity, confidence, and user goal. EvidenceBundle plus rules/templates. CoachFinding cards.
5. Teaching Explain cause/effect with uncertainty and drills. Finding references, drill taxonomy, user profile. Summary, drill plan, follow-up chat context.

Immediate mode

Target under 10 seconds. No broad 3D. Use existing detector outputs, thumbnail-level pose if cheap, simple event timeline, and one coachable observation such as "club started too far inside" only when visible.

Deep dive mode

Can take longer. Run denser frame extraction, pose, club tracking, ball/impact evidence, visual layer generation, metrics, and a richer coaching summary. Use background jobs and persisted run state.

Annotations

The target is Snapchat-like coach markup: bold, readable, scrub-aware overlays that feel alive on video but are grounded in a strict JSON track contract. Use social golf analysis research as pattern inspiration: checkpoint-first, one or two strong marks, cause/effect, and a drill constraint. Keep SwingCoach's visuals and wording original.

Priority

Address shaft plane

Persistent line through shaft at setup. Useful for later comparison against takeaway and delivery.

Priority

Takeaway checkpoint

Clubhead relative to hands and shaft pointing near ball line at lead-arm-parallel.

Priority

Delivery shaft

Transition/delivery shaft line compared with trail forearm or original plane, only when confident.

Priority

Head and hip reference

Address reference lines for posture loss, early extension, sway, and slide checks.

Later

Clubface and wrist

High value but risky from weak video. Do not claim face angle until orientation is actually validated.

Later

3D skeleton replay

Useful for explanation, but only after monocular/multiview accuracy is known for golf-specific poses.

Annotation contract

Keep clean video as the source of truth. Render overlays client-side from JSON tracks with normalized coordinates, event windows, layer names, confidence, source frame references, and an explicit publish/withhold decision. Burned-in MP4 overlays can remain an export feature, not the normal app path.

Metrics

Metrics should start as "publish only when the measurement is honest." A metric that looks precise but is not calibrated will damage trust faster than a missing metric.

V1 safe metrics

tempo phase duration swing window head drift 2D hip depth 2D

These can be useful with clear caveats. They do not require pretending to know absolute 3D joint angles. Express them as visual deltas and categories before exact numbers.

Deep metrics

X-factor pelvis/chest turn side bend club path kinematic sequence

These need calibrated 3D, strong view constraints, or a trained golf-specific model. They are roadmap items, not immediate app claims.

Metric family Feasibility Notes
Tempo and timing Good Requires reliable phase events. Good first feedback because it is temporal rather than geometric.
2D checkpoint deltas Good if view-gated Head drift, hip depth, hand depth, and shaft plane can be useful from DTL/FO with explicit camera assumptions.
Club delivery Hard Club path, face angle, attack angle, and dynamic loft are launch-monitor-grade concepts. Video can estimate proxies, not truth, without validation.
3D body kinematics Possible, validate hard Commercial apps now claim single-video 3D golf data. SwingCoach should benchmark before publishing similar numbers.
Kinematic sequence Deep research True segment velocity sequence needs pelvis, thorax, arm, and club angular velocity. Monocular estimates may be directionally useful but need ground truth.

Computer Vision Ladder

The practical approach is a ladder: start with evidence the phone can gather now, then climb toward richer 3D only when validation proves each rung.

Rung 1: Current on-device detector

YOLO/Core ML object detector for shaft, clubhead, and ball candidate; V2 state machine for addressed ball, club sweep, and ball departure.

Rung 2: Dense offline evidence

Use the same detector at denser sampling for uploaded clips. Add temporal smoothing and strict event windows.

Rung 3: Pose plus club fusion

Run pose landmarks and custom club keypoints. Fuse wrists, shaft, clubhead, ball, and event timings into a versioned evidence bundle.

Rung 4: Golf-specific monocular 3D

Benchmark SAM 3D Body, MediaPipe/BlazePose-style outputs, MotionBERT-like temporal models, Sportsbox/Onform-style claims, and custom golf models against labelled fixtures.

Rung 5: Multi-view and calibration

Two-phone capture or calibrated practice bay. This is closer to OpenCap and TPI-style workflows and is the most credible path for body kinematics.

SAM 3D

Promising for body/object reconstruction from images, but golf swing analysis needs temporal consistency, club/ball handling, and metric validation.

MediaPipe / Apple Vision

Useful for pose landmarks, posture, and event support. Treat world coordinates as model-relative until calibrated.

Custom golf model

Likely needed for club shaft, clubhead, ball, wrists, and golf-specific event detection under motion blur.

Golf Theory Notes

TPI's useful product lesson is not "force everyone into one look." It is "separate WHAT the swing is doing, HOW forces and sequence are produced, and WHY the golfer's body or intent may be creating that pattern."

Kinematic sequence

TPI describes efficient downswing speed transfer as lower body, trunk, arms, then club. The idea is valuable, but SwingCoach should avoid claiming true sequence until it can estimate segment angular velocities reliably.

Swing characteristics

TPI's common characteristics are a useful issue taxonomy: loss of posture, flat shoulder plane, early extension, over the top, sway, slide, reverse spine angle, casting, scooping, chicken wing, and others.

Ball flight laws

TrackMan/Titleist concepts such as face angle, path, attack angle, low point, and dynamic loft are the right language for ball outcome, but most of them are not directly visible from one normal phone video.

Teaching method

The app should prioritize root cause over symptom: setup and takeaway often explain transition and impact compensations. Coach findings should say "fix this first" when confidence allows.

Product principle: compare the golfer to a useful task model, not to a celebrity swing. The best feedback explains an efficient pattern while respecting that different bodies can arrive there through different styles.

Unlimited Resources Version

If resources were not constrained, SwingCoach would train a golf-specific perception and coaching model, not simply glue generic pose models together.

Data engine

Build a multi-view dataset with synchronized iPhones, launch monitor, force plates, optional IMUs, and expert coach labels. Capture DTL, face-on, left/right handed players, many clubs, many surfaces, and bad/noisy examples.

Foundation model

Train a temporal golf model that predicts phase events, body keypoints, club keypoints, ball state, camera calibration, confidence, and coach-relevant embeddings from video.

Metric calibration

Distill lab-grade signals into phone-visible proxies. Learn which metrics can be safely estimated from one view, which need two views, and which should be withheld.

Coaching intelligence

Train/evaluate a finding ranker using coach labels and longitudinal outcomes: what changed after a drill, what made the next swing better, and what feedback caused confusion.

Model What it would do Why it matters
Swing foundation model Temporal encoder over video frames, object tracks, pose, audio, and metadata. One model can share context across detector, events, metrics, and coaching.
Golf club keypoint model Predict grip, hands, shaft endpoints, clubhead, face proxy, and blur-aware confidence. The club is the object generic pose systems ignore, but coaches care about most.
Camera calibration model Infer view quality, target line, horizon, lens effects, scale cues, and when not to measure. Prevents false precision and guides users to capture better video.
Coach finding ranker Map evidence to priority, drill, and explanation style using expert labels. Turns analysis from "detected flaw list" into a lesson plan.

Roadmap

This sequence aims to get one narrow coaching path right before expanding the system.

M0: Device smoke and cleanup

Build to phone, verify detector model loads, record/import a few swings, verify analysis upload still works, and log current UI pain points. Add capture-quality guidance and retake prompts for bad framing, missing ball, unstable camera, or wrong view.

M1: Evidence bundle contract

Define versioned JSON for events, layers, confidence, source frames, and generated findings. Add tests with empty and populated tracks.

M2: Annotation V1

Generate address shaft plane, takeaway checkpoint, delivery shaft, head/hip reference, and phase markers. Validate visually on fixtures.

M3: Coaching Finding V1

Choose one or two findings from high-confidence annotations. Pair each with evidence, uncertainty, user goal/profile context, and a drill from an expanded taxonomy.

M4: Metrics V1

Publish safe timing and 2D checkpoint metrics. Keep deeper body/club delivery metrics behind research flags.

M5: Practice loop

Let users compare before/after swings against the same finding and mark whether a drill helped. Capture outcome feedback so the coaching ranker can improve over time.

Release gate Acceptance criteria before enabling generated coaching broadly
Device matrix Smoke on at least one older supported iPhone and one current iPhone: model loads, capture remains responsive, imported trim detection runs, async analysis completes.
Detector fixtures Track expected count, false positives, false negatives, and impact timing across DTL/FO, left/right handed, mats/grass, bright/dim light, multiple balls, and no-swing hard negatives.
Overlay review Every generated layer needs visual fixture review: no one-frame flashes, no jumping geometry, source-frame references, and clear withhold behavior when confidence drops.
Coach review A human coach should approve the first finding taxonomy and reject misleading claims before they reach users.
User context Collect optional goal, handicap/skill level, pain or mobility constraints, club, and practice setting before making personalized drill recommendations.
Privacy and consent Separate analysis upload consent from training-data consent. Document retention, deletion, and whether videos may be used to improve models.

General Cleanup

The cleanup should support the coaching pipeline rather than become an abstract refactor. The highest leverage work is around boundaries, state persistence, and validation.

iOS architecture

Audit shared singletons, tab handoff state, analysis persistence, retry flows, and model loading failure states.

Backend reliability

Persist analysis runs, make jobs resumable/cancellable, and separate artifact generation from coaching generation.

Validation harness

Add fixture-level visual tests for every generated overlay before re-enabling automatic annotations.

Capture guidance

Detect bad framing, wrong view, missing ball, unstable camera, and poor lighting before analysis spends time on unusable video.

User constraints

Store optional goal, club, skill level, injury or mobility notes, and practice setting so drills do not become generic advice.

Privacy and consent

Separate upload for analysis from opt-in training consent, and make deletion/retention rules visible before dataset collection scales.

Docs

Keep this plan linked from docs. Update it when pipeline behavior changes.

Dataset hygiene

Track source timing, view, handedness, club, surface, ball visibility, and labels for detector training videos.

Product language

Use confidence-aware text. Say "visible from this view" and "not enough evidence" often.
