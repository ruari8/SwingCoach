Documentation-only reference

Make the phone-video coach real, without pretending the hard parts are solved.

SwingCoach already has a serious product shell: iOS capture, library, trimming, experimental auto-detection, async backend runs, R2 artifacts, playback, and reset-state coaching. The missing product is the actual evidence engine: trustworthy events, body and club tracking, metrics, annotations, drill mapping, and a teaching voice that proves its claims on video.

iOS-first confidence-gated single-phone MVP hybrid future

Read This First

This plan is intentionally candid. It separates product infrastructure from coaching intelligence, because those are not at the same maturity level. The app can capture and route swings today. It cannot yet make high-confidence biomechanical claims from one phone video without a rebuilt detection, annotation, and metric contract.

Product thesis

The valuable loop is not "upload a clip and receive a paragraph." It is: find a real swing, show the important frames, draw only defensible overlays, explain one priority, prescribe a useful drill, and compare the next rep against the same evidence.

Truth standard

Every automated claim needs a source, confidence, and fallback. If shaft, ball, or body evidence is weak, the product should say less and show the uncertainty instead of decorating the video with false precision.

Best next move

Rebuild the annotation contract around a small set of reliable overlays, then re-enable metrics only when each metric has a detector source, calibration assumption, confidence threshold, fixture proof, and UI copy.

Core Holes Before A Full Coaching Pipeline Is Real

The biggest hole is not UI. It is the absence of a validated evidence graph from video to coaching claim. The app needs to know what happened, where it happened, how confident it is, and which visual proof supports it.

1. Event model

Define and detect address, takeaway, top, transition, shaft-horizontal positions, impact proxy, early follow-through, finish, and end. Do not expose a metric until it declares which event timestamps it depends on.

2. Coordinate system

Normalize source timeline, frame coordinates, view type, handedness, camera roll, lens/focal hints, body scale proxy, and confidence. The current track envelope is a good place to formalize this.

3. Club and ball identity

A generic ball detector is not enough. The product needs addressed-ball identity, shaft/clubhead continuity, and ball departure confidence tied to the same locked strike patch.

4. Body tracking confidence

Pose quality needs per-joint confidence, temporal smoothness, occlusion handling, and view-specific validity. Face-on and down-the-line do not support the same claims.

5. Metric publication rules

Each metric needs a "publish / omit / describe qualitatively" rule. Numeric output should be reserved for measurements with enough calibration and repeatability.

6. Product memory and personalization

The coach needs user profile, goals, handicap, physical constraints, equipment, recent drills, previous swing patterns, and whether a suggested feel actually improved the target evidence in later sessions.

How We Use A Detected Swing

Detection is a routing primitive first, not a full diagnosis. It should preserve practice flow, reduce trimming work, and provide a structured seed for deeper analysis.

1
Find candidate swing windows. V2 reports start, impact proxy, end, confidence, and evidence. Capture/Trim use those ranges as editable suggestions, not immutable truth.
2
Attach detector evidence to the clip. Store candidate trace summaries: address lock, patch departure, club sweep, ball inventory change, timing, and sampling cost. This becomes the first evidence object for backend analysis.
3
Pick the feedback lane. If the user is practicing live, return a fast cue from robust evidence. If the user requests analysis, upload the clip and run the deeper backend pipeline.
4
Generate evidence tracks. Backend or on-device analysis converts the swing into a timeline: events, pose tracks, club tracks, ball/strike evidence, quality flags, and annotations.
5
Coach only from confirmed evidence. Select one or two priorities, render overlays that prove them, return drills, and save the analysis for comparison against future swings.

Immediate Feedback Vs Deep Dive Analysis

The competitive product loop is likely fast, on-device, and voice-forward. The research loop can be slower, richer, and backend-heavy. SwingCoach should support both without forcing them into one pipeline.

Metrics And Golf Theory

Use golf theory as a scaffold, not as permission to over-measure. TPI-style kinematic sequence and launch-monitor metrics are useful targets, but a single uncalibrated phone camera cannot honestly reproduce a motion-capture lab, radar launch monitor, force plates, or pressure plates.

TPI and kinematic sequence

TPI describes efficient downswing sequencing as proximal-to-distal energy transfer: pelvis, trunk/ribcage, arms, then club, with each segment generally peaking faster and later than the previous segment. This is a good long-term coaching target, but true segment rotational velocities are a 3D measurement problem.

References: TPI kinematic sequence basics, TPI kinematic sequence revisited.

Biomechanics caution

A systematic review of golf-swing biomechanics notes that kinematic methods vary, and common parameters such as X-factor and kinematic sequence are sensitive to methodology. Product metrics should expose method and confidence.

Golf swing biomechanics systematic review

Launch-monitor caution

TrackMan-style club data such as club path, attack angle, face angle, dynamic loft, and club speed are defined around impact and club geometry. Phone video can estimate proxies, but should not label them as launch-monitor equivalents without validation.

Computer Vision Plan

Build the CV stack in layers. The MVP should be interpretable and sparse. The long-term path can use heavier models, but only after the data contract makes failures visible.

Stage Technique Use in SwingCoach Feasibility / risk
Pragmatic MVP YOLO/Core ML object detection, addressed-ball patch watcher, club evidence, temporal state machine. Find swings, lock strike patch, create editable trim ranges, detect obvious practice vs strike cases. Feasible and already underway. Risk is domain diversity and pseudo-label quality.
2D body layer Apple Vision body pose or MediaPipe BlazePose-style landmarks. Primary golfer, joints, posture, head box, hands, gross rotation proxies. Feasible, but club occlusion and fast motion require confidence gating.
Track smoothing Kalman/OneEuro smoothing, optical flow, point tracking, interpolation between detections. Stable club trails, hand path ribbons, less flickering overlays. Feasible. Risk is smoothing away real transition/impact timing.
Segmentation assist SAM3 or similar promptable segmentation offline. Dataset labeling, shaft/clubhead/ball pseudo-labels, visual QA, hard-case discovery. Good as a labeling aid. Too heavy and brittle for live iPhone detector in current form.
Monocular 3D Temporal 2D-to-3D pose lifting, MotionBERT-like motion priors, Apple Vision 3D where available. Qualitative 3D replay and experimental pelvis/chest turn proxies. Useful, but depth ambiguity and occlusion make precise coaching claims risky.
Multi-view Two phones or phone plus fixed camera, calibration, triangulation. Reliable 3D body and club path, better kinematic sequence, better validation dataset. Technically strong, product-friction high. Good for power-user or lab mode.
Club and face modeling Specialized clubhead/shaft keypoints, temporal transformer, synthetic data, high-FPS supervision. Club path, shaft plane, impact proxy, possibly face orientation under constraints. High value and high risk. Face angle/dynamic loft need much more than a generic detector.
Monocular 3D rule: use it for visualization, priors, and relative comparison first. Treat precise 3D biomechanics, kinematic sequence, and launch-monitor-like metrics as research until validated against ground truth.

Unlimited Resources Model Training Ideas

If resources were unconstrained, the winning system would combine real capture data, lab-grade ground truth, coach labels, synthetic augmentation, and on-device distillation.

Dataset program

  • Collect paired DTL and face-on iPhone video across clubs, skill levels, lighting, indoor/outdoor, left/right handed golfers, mats, grass, nets, and crowded ranges.
  • Record synchronized lab ground truth: optical motion capture, launch monitor, pressure plates, force plates, high-speed club/ball cameras, and calibrated multi-view video.
  • Split by golfer/session/course, not adjacent frames, to avoid optimistic validation.
  • Store uncertainty labels: occluded, low light, shaft blur, ball hidden, no ball, practice swing, background strike, camera moved.

Label taxonomy

  • Events: address, takeaway, top, transition, impact proxy, finish.
  • Objects: ball candidates, addressed ball, clubhead, shaft endpoints, clubface proxy.
  • Body: joints, segments, visibility, occlusion, 3D ground truth.
  • Coaching: faults, priorities, drill mappings, no-call cases.

Model family

  • On-device object/keypoint model for club, shaft, ball, and strike patch.
  • Temporal event transformer for swing phases and confidence.
  • Motion model for 3D body and club priors.
  • Teacher ensemble on backend, distilled student for iPhone.

Evaluation suite

  • Fixture tests for exact swing count, no false positives, event timing, left-handed support, background bay noise, spare balls, bad framing, glare, slow motion, and no-ball rehearsals.
  • Metric repeatability tests: same swing, same camera, different random seed; same swing, slight camera shift; same golfer across days.
  • Coach-output tests: no unsupported claims, uncertainty preserved, drill matches detected priority, user-visible explanation cites video evidence.

Decision Points

Decision Options Recommendation
Where should first real analysis run? Backend-only, on-device-only, hybrid. Hybrid architecture, backend implementation first. Keep contracts portable to on-device.
What metric ships first? Tempo, shaft plane, posture, kinematic sequence, launch-monitor proxies. Tempo and event-based measures first. Then DTL plane/path visuals. Save sequence and launch metrics for validation.
Should generated overlays be baked into MP4? Baked video, declarative app-rendered overlays, both. Declarative tracks as source of truth; baked video as compatibility/share artifact.
How strict should confidence be? Always show, hide low-confidence, show uncertainty. Hide misleading geometry and show uncertainty summaries. Confidence is a user-trust feature.
Should face-on be equal priority? Equal support now, DTL first, user-selected view packs. DTL first for current product maturity. Face-on next once view-specific metrics and capture guidance are defined.
How to handle TPI-style claims? Market as biomechanics, research-only, validated premium tier. Use TPI concepts as roadmap and education. Do not publish numeric sequence claims until ground-truth validated.
