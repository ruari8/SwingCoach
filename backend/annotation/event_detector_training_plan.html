Event Detector Training Plan

A practical plan for turning 500+ manually corrected P1-P10 swing labels into a video event detector that finds coaching checkpoints for DTL annotations.

Why Not Train On Only The Ten Frames?

Ten-frame classifier

You could train a still classifier: 

input = one labelled frame
label = P1 or P2 or ... P10

This teaches the model what a selected P-position looks like, but it does not teach the model how to find that position inside a full swing.

Video event detector

Production gives us a video. The model has to search through the sequence and find the moments. 

input = all sampled frames
label = probability curve for each P

This also helps with ambiguous stills. P3 and P5 can look similar; motion direction separates them.

What "Fixed Frame Count" Means

Neural networks train in batches. Batches are easiest when every input has the same tensor shape. That does not mean the original videos must have the same length, FPS, resolution, or slow-mo speed.

It means we create a model input by sampling each swing into a fixed number of frames. 

video A: 1.6s normal speed -> sample 160 frames
video B: 8.0s slow-mo      -> sample 160 frames
video C: 2.4s normal speed -> sample 160 frames

The timestamp labels are converted into positions on that sampled timeline. We keep the mapping back to original frame/time for output.

Alternative

We can support variable-length videos using padding and masks. That is common too, but fixed sampling is simpler for the first model and usually good enough for trimmed swing clips.

Resolution is also normalized for the model, usually something like 224x224 or 256x256. The original video stays untouched for rendering annotations.

How The Model Learns Events

Hard timestamp becomes soft target

Your label says: P6 = frame 96. During training we do not only mark one frame as right and every neighboring frame as completely wrong. We create a small soft peak around the label.

This matters because P6/P7 can be separated by one frame in low-FPS video, and sometimes the best visible frame is adjacent to the true mechanical instant.

Model Options

Option Input How It Works Pros Cons Recommendation
Still-frame classifier Individual frames Classifies each frame as P1/P2/etc or no-event. Easy to understand. Weak motion context; P3/P5 ambiguity; not ideal for event localization. Useful only as a teaching baseline.
Pixel video model Sampled video frames Video model learns visual and temporal features from pixels. Clean concept; no manual feature engineering. Needs transfer learning and careful compute choices with 500 videos. Good candidate if we use a pretrained video backbone.
Frame encoder + temporal head Sampled frames Pretrained image encoder converts each frame into a feature vector; TCN/LSTM/Transformer finds P peaks. Data-efficient; easy to debug; practical first real model. May miss motion blur cues unless temporal head is strong. Best first training path.
Feature-based temporal model Pose/YOLO tracks Uses body/object coordinates instead of raw pixels. Interpretable; smaller model. Depends on detectors; shaft/club can fail in blur. Useful as an auxiliary model or confidence signal.
Hybrid model Pixels + pose + YOLO Combines learned visual features with golf-specific tracks. Likely strongest eventually. More moving parts. V2 after pixel/event baseline works.

Recommended First Training Architecture

V1: pretrained frame encoder + temporal head

full swing video
-> sample 160 frames
-> resize/crop frames to model input size
-> pretrained frame encoder
-> feature vector per sampled frame
-> TCN or small Transformer over time
-> 10 event heatmaps
-> P1-P10 peaks/windows

This keeps the user-facing data simple: videos plus timestamps. The "features" are learned automatically by the encoder.

Why this first?

  • 500 videos is enough to fine-tune a small head, but may be small for training a video model from scratch.
  • Transfer learning lets us reuse a model that already understands edges, bodies, objects, motion, and visual structure.
  • TCNs/Transformers over frame features are easier to inspect than a giant end-to-end video model.

Modern CV Landscape As Of July 2026

The current broad pattern is not "train a CNN from scratch on 500 golf swings." It is transfer learning from large pretrained vision/video models, then fine-tune a smaller head for the specific task.

Area Modern Families Why It Matters Here
General image representation ViT, DINOv2/DINOv3, ConvNeXt V2, EVA-02, SigLIP/SigLIP 2 Good frame encoders. They turn a frame into useful learned features.
Video understanding VideoMAE/VideoMAE V2, Video Swin, Hiera, InternVideo2, long-video MAE variants Good candidates when we want the model to learn temporal motion directly from video.
Object detection YOLO family, RT-DETR, DETR/DINO-style detectors Useful for ball, clubhead, shaft, and later annotation geometry.
Pose estimation ViTPose, RTMPose, RTMO, MMPose ecosystem Useful auxiliary signals for P3/P4/P5/P6 and body-reference annotations.
Segmentation/tracking SAM 2 and video segmentation models Useful later for club/shaft masks, but likely too heavy for the first event detector.

Training Pipeline

1. Prepare manifest

One JSON/CSV row per swing, with video path, metadata, and P1-P10 timestamps. The timestamps are the source of truth.

2. Split by source

Train/val/test split by YouTube video, range session, or original source. Do not let near-duplicate swings leak across splits.

3. Sample frames

Convert every video into a fixed number of sampled model frames while preserving mapping back to original time and frame index.

4. Build heatmaps

Convert each P timestamp into a soft target curve over the sampled frames.

5. Train model

Fine-tune the temporal head first. Optionally unfreeze the encoder later if validation error plateaus.

6. Evaluate

Measure milliseconds/frame error per P-position, especially P6/P7, and inspect failure sheets for the worst cases.

What To Ignore For Now

We do not need to solve every ML question before starting. Do not start with full hybrid pose+YOLO+pixels unless the pixel/temporal baseline fails. Do not spend weeks perfecting a contact-sheet review flow if labels are already corrected. Do not train from scratch. Do not let annotation geometry leak into the event detector milestone.

Model starting point

Start with a frozen pretrained frame encoder and a lightweight temporal head. Candidate encoders: DINOv3/DINOv2, ConvNeXt V2, or a VideoMAE family model if the runtime makes video backbones easy.

Candidate temporal heads: TCN first for simplicity, small Transformer if we need more global sequence reasoning.
